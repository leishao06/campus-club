## 2. 背景与问题陈述

### 2.1 业务背景

高校学生社团招新存在典型痛点：**信息不对称**（简章空泛、真实体验难获取）、**报名流程繁琐**（多渠道重复填表、进度不透明）、**新生决策成本高**（难以快速匹配兴趣与精力约束）。校方需要招新过程**可管可控**与**数据可视化**，以支撑社团生态治理与评优。

### 2.2 产品定位

面向高校招新季的 **「兴趣发现 + 智能匹配 + 报名录取 + 校方治理」** 一体化平台，覆盖新生从 **入学前预热 → 兴趣测评 → 匹配选社 → 报名考核 → 入社留存** 的全旅程，并为社团与校方提供数字化招新与监管能力。

### 2.3 目标用户

| 角色 | 说明 |
|------|------|
| 新生（C 端） | 本科/研究生新生为主，需低门槛完成选社与报名 |
| 社团管理员 | 社长、招新负责人、内容运营等 |
| 校方/社联管理员 | 团委、学工、社团联合会等监管与运营角色 |

## 3. 目标与成功指标

### 3.1 定性目标

- **新生侧**：降低选社决策成本，**3 分钟内**完成兴趣匹配闭环（以「极速测评 + 看到推荐列表」为完成定义）；报名流程相对传统方式效率显著提升。
- **社团侧**：降低线下摆摊依赖，实现 **报名—筛选—录取** 流程线上化与自动化，减轻招新期人工整理与沟通成本。
- **校方侧**：社团招新全流程可审计、数据可分析，形成权威信息发布渠道与正向生态运营依据。

### 3.2 量化指标（建议与基线对照）

> 以下指标需在试点前建立**基线**（问卷/工时抽样/对照组），否则难以对外归因。

| 维度 | 指标示例 | 说明 |
|------|----------|------|
| 新生 | 极速测评完成率、P90 完成时长≤3 分钟、推荐列表 CTR、投递转化率、录取后 30 日活跃/NPS | 「匹配度」可辅以问卷或行为 proxy |
| 社团 | 招新相关周工时、人均处理报名表时间、录取操作次数、摆摊人天/物料费（试点前后） | |
| 校方 | 审核时效、投诉闭环时长、看板使用率、数据导出次数 | |
| 平台 | 系统可用性、峰值并发、关键接口 P95 | |

## 4. 产品范围

### 4.1 本期包含（MVP 建议边界）

- 账号与身份：对接校方统一身份认证（SSO）或校方指定账号体系；角色区分新生、社团管理员、校方管理员。
- **双模式测评**（极速 30 秒 + 深度约 2 分钟）与 **匹配结果展示**（TopN + 匹配理由，可解释）。
- **社团标准化主页**（六大模块，见 6.2）与招新发布、校方审核。
- **一键多投通用档案**、投递上限、**报名进度追踪**、基础消息通知。
- 社团端：**报名表管理、筛选、录取通知**；可选 **线上笔试/面试排期**（按排期分阶段上线）。
- 校方端：**招新时间窗、社团资质与年审、内容审核、投诉工单、基础数据看板**。

### 4.2 本期不包含或后置

- 与校外第三方深度社交绑定；跨校数据互通。
- 复杂 OA 多级审批（可先轻量审核流）。
- 重度 AI 生成内容；完整 RAG 问答机器人（可作为 P1）。
- 法律效力外的「电子合同」类能力（电子入社函以校内行政约定为准）。

## 5. 用户故事与核心场景

### 5.1 新生

- 作为新生，我希望 **30 秒完成极速测评** 并立即看到推荐社团，以便快速缩小选择范围。
- 作为新生，我希望看到 **匹配理由与社团真实信息**，以便建立信任并减少踩坑。
- 作为新生，我希望 **只填一次档案即可投递多个社团**，并 **实时看到进度**，减少反复询问。
- 作为新生，我希望用 **筛选与对比** 辅助决策，并对内向者提供 **匿名咨询** 等低压力沟通方式。

### 5.2 社团

- 作为社团管理员，我希望 **标准化搭建招新主页** 并通过校方审核后发布，减少重复答疑。
- 作为社团管理员，我希望 **集中查看投递、筛选、批量录取**，并可选 **线上笔试与面试预约**。
- 作为社团管理员，我希望看到 **招新数据看板**，优化招新策略。

### 5.3 校方

- 作为校方管理员，我希望 **统一管理招新季规则、社团资质与违规处置**，并 **可视化关键数据** 支撑治理与评优。

## 6. 功能需求

### 6.1 新生端（C 端）

#### 6.1.1 入学前预热与招新季

- **招新季日历**：展示全校招新起止时间、重要节点（宣讲、笔试集中日）；支持订阅提醒。
- **官方资讯区**：招新政策、合规指南、避坑提示；与校方「官方发布」模块同步。
- **未测评用户**：首页强引导「极速测评」；深度测评作为进阶入口。

#### 6.1.2 兴趣测评与智能匹配

| 子功能 | 需求描述 | 优先级 |
|--------|----------|--------|
| 极速测评（约 30 秒） | 3～5 道选择题：每周可投入时长、入社核心目标（学技能/社交/学分/评奖等）、基础兴趣领域、门槛接受度（面试/笔试）等；完成后进入匹配结果 | P0 |
| 深度测评（约 2 分钟） | 补充特长、经历、职业规划、性格倾向、时间预算等；刷新匹配方案 | P0 |
| 测评状态 | 未开始/极速完成/深度完成；支持按校方规则在招新季重置或每学期更新 | P1 |
| 匹配结果页 | 展示 Top10（可配置）匹配社团；每个社团展示 **匹配度展示方式**（建议区间或星级 + 文案，弱化绝对分数争议）+ **匹配理由**（可解释，非黑盒） | P0 |
| 硬条件反馈 | 可选展示「因不接受面试/时间不足等被过滤」的提示与「放宽条件」二次浏览 | P1 |

**匹配理由示例（产品需提供模板库）**：  
「匹配度 92%｜你希望提升新媒体技能，该社团有系统运营培训，社员多次获校级新媒体奖项；每周活动约 2.5 小时，符合你的时间预期。」

#### 6.1.3 发现页与推荐

- **千人千面首页**：综合测评结果、浏览/收藏/投递行为动态调整推荐；加入 **多样性控制**（同品类连续曝光上限、新社团冷启动流量等，参数可由校方配置）。
- **分类导航**：学术科技、文艺体育、志愿公益、创新创业、兴趣爱好等官方分类；支持按匹配度、热度、招新截止时间、零基础友好等排序。
- **筛选器**：无面试门槛、零基础友好、时间投入≤X 小时/周、第二课堂学分、无会费、无会费等 **标签化一键筛选**（标签与社团填报字段绑定）。
- **搜索**：社团全称/简称/别名、标签、部门名、关键词；维护同义词库。

#### 6.1.4 社团信息透明化（标准化主页）

发布招新前 **六大模块必填**（缺一不得正式发布，可存草稿）：

| 模块 | 内容要求 |
|------|----------|
| 官方档案 | 成立时间、主管单位、规模、核心荣誉、年审结果 |
| 招新简章 | 人数、部门、要求、考核方式、报名截止时间、会费；**会费须经校方审核通过后展示** |
| 真实动态 | 活动图文/视频、预告、社员 vlog、宣讲回放等 |
| 社员口碑 | 实名/匿名评价 + 标准化标签（氛围、学习成长、时间占用、任务强度等）；恶意评价过滤与申诉 |
| 常见 QA | 高频问题与答复 |
| 官方避坑提示 | 社联合规提示、违规记录公示（若有） |

**特色功能**

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 云逛社团 | 360° 活动室全景、招新直播回放；存储与有效期策略可配置 | P1 |
| 匿名咨询 | 新生匿名提问，问答可选同步展示在主页；敏感词与防骚扰 | P1 |
| 社团对比 | 最多 3 个社团对比：时间、门槛、会费、口碑、特色等 | P1 |

#### 6.1.5 报名与考核闭环

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 通用档案 | 标准化字段（姓名、学号、联系方式、兴趣特长、自我介绍等）+ 社团自定义附加题；**一份档案多投** | P0 |
| 投递上限 | 默认最多 5 个（可配置）；需明确：是否含已拒绝后补投、是否区分志愿序 | P0 |
| 进度追踪 | 节点可视化：已投递→已查看/已受理→笔试/面试→待录取→已录取/已拒绝；**「已查看」若技术无法保证，则改为诚实表述如「已受理」** | P0 |
| 消息通知 | 关键节点站内信 + 微信服务通知（模板清单另附）；支持订阅开关与静默时段 | P0 |
| 线上笔试 | 题型：单选/多选/简答；限时、自动收卷；简答题阅卷队列与 SLA（P1） | P1 |
| 面试预约 | 社团排班、新生预约、改期规则、冲突提示（多社团） | P1 |
| 电子入社函 | 录取后生成电子凭证；与社员管理/社群对接策略见非功能与集成；失败时工单补偿 | P1 |

#### 6.1.6 入社后（轻量）

- 入社引导、活动日历入口、退社/转部门规则说明（链接校方制度）。
- 若对接二课/学分：展示对接状态与说明（可选）。

#### 6.1.7 反馈与负向行为

- 对推荐结果：「不感兴趣」及原因（已加入其他社、内容不准等）用于优化排序与内容。
- 投诉入口：跳转校方投诉工单（见 6.3）。

---

### 6.2 社团管理端

#### 6.2.1 入驻与主页

- 入驻申请、资质更新、年审报备；**仅审核通过可获得招新发布等权限**。
- 标准化主页模板；招新标签、物料上传、宣讲直播预约。
- **角色权限**：社长、招新负责人、内容运营、财务（会费相关）等细分（RBAC）。
- 简章重大变更可要求重新审核；历史版本留痕。

#### 6.2.2 精准招新

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 定向触达 | 按标签筛选潜在新生；**默认不直接暴露联系方式**；平台内邀约 + 频次上限，防骚扰 | P1 |
| 主动邀约 | 话术模板、每日上限；新生可拒收营销类推送 | P1 |

#### 6.2.3 报名与考核

- 自定义报名表字段；投递列表汇总、条件筛选、批量标记、导出（权限与脱敏由校方策略约束）。
- 线上笔试创建、面试时段排班、批量录入结果、一键发送录取/拒绝通知（模板化）。
- 录取批次：提前批、补录、候补（规则可配置）。

#### 6.2.4 社员与数据

- 社员电子档案：入社时间、部门、活动参与、获奖（最小集）。
- 招新数据看板：投递量、录取率、院系/性别/匹配度分布等（字段依隐私合规收敛）。

---

### 6.3 校方 / 社联监管端

#### 6.3.1 社团全生命周期

- 入驻、年审、注销线上审核；违规处罚：**警告 → 限制 → 下架招新 → 注销** 等阶梯策略与申诉期。
- 仅合规社团可发布招新与展示会费。

#### 6.3.2 招新监管

- 全校招新起止时间、会费标准报备与审核。
- 内容审核：招新物料、动态、评价；敏感词与图片审核策略。
- **投诉处理**：新生一键投诉、分配、处理、留痕、闭环统计。

#### 6.3.3 数据与报告

- 全校看板：社团总量、招新人数、分类分布、报名率/入社率、活跃度、评价趋势等。
- 多维度导出：**水印、操作人、用途说明**；敏感维度（如民族、生源地）仅在校方授权下提供且默认脱敏聚合。

#### 6.3.4 官方信息发布

- 招新政策、合规指南、违规公示、评优结果等 **唯一权威出口** 与推送策略。

#### 6.3.5 系统与审计

- 操作审计日志：关键审核、导出、处罚、规则变更。
- 管理员角色细分：超级管理员、内容审核员、数据只读等（可配置）。

## 7. 智能匹配逻辑（产品层说明）

采用 **三层匹配模型**，兼顾精准度、冷启动与可解释性；**权重与阈值支持校方在授权范围内配置**，避免完全黑盒。

### 7.1 第一层：规则过滤（硬匹配）

- 依据新生明确约束排除不满足项：如不接受面试、每周可投入时长上限、不接受会费等。
- **规则优先级与冲突处理**需在配置表中定义并版本化。

### 7.2 第二层：标签相似度 + 协同过滤（软匹配）

- **标签相似度**：新生画像标签与社团分类/招新/内容/口碑标签做相似度计算，建议基础权重约 **70%**（可调）。
- **协同过滤**：基于「同行为用户偏好」补充推荐，缓解冷启动，建议权重约 **30%**（可调）；冷启动前期可加大内容标签与运营种子权重。

### 7.3 第三层：个性化动态排序

- 根据浏览、收藏、点赞、咨询、投递行为动态调整权重。
- **多样性控制**：品类打散、探索流量、新社团保护期。
- **负反馈**：不感兴趣、低评分内容降权。

### 7.4 算法治理

- 匹配分展示策略、文案模板库、定期校准（防分数通胀）。
- 公平性：避免基于敏感属性的歧视性规则；推荐解释避免不当标签。

## 8. 关键业务流程与状态

### 8.1 报名单状态机（示例）

```
草稿 → 已投递 → 已受理 → 考核中（笔试/面试）→ 待录取 → 已录取 / 已拒绝 / 已撤回
```

- 「已受理」定义须与研发能力一致；撤回规则、超时自动失效需在业务规则中写明。

### 8.2 社团招新发布

```
草稿 → 待校方审核 → 已发布 / 驳回 → （修改）→ 待审核
已发布 → 招新结束归档
```

### 8.3 投诉工单（示例）

```
已提交 → 处理中 → 已结案；可分支：升级、驳回
```
